## Overview

pfSense is used as the **network perimeter firewall and traffic control system** within this SOC environment. It is responsible for monitoring, filtering, and controlling network traffic between internal systems and external networks.

To enhance detection capabilities, pfSense is extended with **Suricata IDS/IPS**, enabling real-time intrusion detection at the network level.

---

## Key Responsibilities

* Network traffic filtering and firewall enforcement
* Network segmentation and rule-based access control
* Intrusion Detection System (IDS) monitoring via Suricata
* Logging of network events and suspicious activity
* Blocking malicious IP addresses and traffic patterns

---

## Architecture Role

pfSense operates at the **network boundary layer** of the SOC:

```id="pfsense-flow"
External Network / Internet
          │
          ▼
      pfSense Firewall
   (Traffic Filtering + IDS)
          │
          ▼
   Internal SOC Environment
```

Suricata runs directly on pfSense to inspect traffic in real time.

---

## Suricata IDS Integration

Suricata is deployed as an **intrusion detection system (IDS)** within pfSense.

### Capabilities:

* Real-time packet inspection
* Signature-based threat detection
* Detection of known attack patterns
* Alert generation for suspicious traffic
* Protocol analysis (HTTP, DNS, TLS, etc.)

---

## Detection Examples

Suricata can identify:

* Port scanning activity
* Brute force attempts
* Known exploit signatures
* Malicious outbound connections
* Suspicious DNS queries

When detected, events are logged and can be forwarded into the SOC pipeline.

---

## Logging & Alerting

Security events generated by pfSense and Suricata are:

1. Logged locally within pfSense
2. Forwarded to **Wazuh SIEM**
3. Processed alongside endpoint and system logs

This ensures network-level visibility is combined with endpoint-level detection.

---

## Integration with SOC Workflow

pfSense integrates into the broader SOC pipeline:

* **Suricata → Wazuh**
  Network alerts are ingested into the SIEM for correlation

* **Wazuh → n8n**
  Alerts trigger enrichment and automation workflows

* **n8n → pfSense**
  Automated response actions can push firewall rules (e.g. IP blocking)

* **MISP correlation**
  Suspicious IPs or domains can be matched against threat intelligence

---

## Automated Response Capabilities

pfSense can be used as an enforcement layer for automated response actions.

### Examples:

* Blocking malicious IP addresses
* Restricting access from flagged regions or hosts
* Applying temporary firewall rules during incidents

These actions may be triggered automatically via **n8n workflows**.

## Key Benefits

* Network-level visibility into all traffic
* Early detection of malicious activity before endpoint compromise
* Real-time intrusion detection with Suricata
* Strong enforcement layer for automated blocking
* Complements endpoint-based detection (Wazuh)

---

## Limitations

* Suricata requires tuning to reduce false positives
* Encrypted traffic limits deep inspection
* Firewall rules must be carefully managed to avoid disruption
* IDS alerts require correlation for full context

---

## Design Approach

This implementation follows a layered security model:

* **Prevent** → pfSense firewall rules block unwanted traffic
* **Detect** → Suricata identifies suspicious network behaviour
* **Correlate** → Wazuh links network + endpoint activity
* **Respond** → n8n triggers automated containment actions

This ensures detection and response happens at multiple layers.

---

## Summary

pfSense, extended with Suricata IDS, provides the **network security foundation** of the SOC.

It enables:

* Real-time intrusion detection at the network layer
* Traffic filtering and segmentation
* Integration into automated SOC response workflows

This strengthens the SOC by ensuring threats can be detected and blocked **before reaching internal systems**.
