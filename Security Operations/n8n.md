## Overview

n8n is used as the **automation and orchestration layer** within this SOC. It functions as a lightweight **SOAR (Security Orchestration, Automation, and Response)** platform, enabling automated handling of security alerts generated by Wazuh.

Its primary role is to reduce manual investigation effort by enriching alerts, applying decision logic, and triggering response actions — all without analyst involvement for routine events.

---

## Key Responsibilities

* Ingest alerts from Wazuh via webhook
* Enrich alerts with threat intelligence
* Apply decision logic to determine severity and response
* Automate incident creation in DFIR IRIS
* Trigger response actions (e.g. blocking IPs, disabling accounts)

---

## Architecture Role

n8n sits between detection and response, acting as the central decision-making engine:

```
     Wazuh SIEM
 (Alert Generation)
        │
        ▼
        n8n
 (Enrichment & Decision Engine)
        │
        ├──────────────► DFIR IRIS (Incident Creation)
        │
        ├──────────────► OPNsense (IP Blocking)
        │
        ├──────────────► MISP (IOC Updates)
        │
        └──────────────► Microsoft Entra (User Response)
```

## Infrastructure

n8n runs on a dedicated **VMware Workstation VM** as a **Docker container**, accessed via its web interface for workflow management, monitoring, and execution history.

---

## Workflow Design

The core workflow follows a structured pipeline from alert ingestion through to response.

### 1. Alert Ingestion

Alerts are received from Wazuh via a **webhook trigger node**.

The JSON payload includes:

* Source IP address
* Username (where applicable)
* Wazuh rule ID and description
* Severity level (rule level 1–15)
* Agent hostname
* Timestamp

---

### 2. Data Extraction & Normalisation

Relevant fields are extracted using **Set** and **Code** nodes. Data is normalised into a consistent structure before processing, ensuring all downstream steps receive clean, predictable input regardless of the original log source.

---

### 3. Threat Intelligence Enrichment

n8n queries multiple threat intelligence sources in parallel to build context around each alert:

| Source | Data Retrieved |
|--------|----------------|
| **VirusTotal** | IP / URL / hash reputation and detection ratio |
| **AbuseIPDB** | Abuse confidence score, country, ISP |
| **MISP** | Internal IOC match and associated threat context |

Based on enrichment results, each indicator is classified as:

* **Malicious** – confirmed match or high confidence score
* **Suspicious** – borderline score or partial IOC match
* **Benign** – no matches, low risk score

### 4. Decision Logic

Conditional (IF) nodes evaluate enriched data and route the alert to the appropriate action:

* If IP is confirmed malicious → escalate severity, trigger response
* If login is successful from a malicious IP → disable account and revoke sessions
* If score is below threshold → log and close without action
* If false positive indicators are present → downgrade and close the alert

### 5. Incident Creation

Enriched alerts that meet severity thresholds are forwarded to **DFIR IRIS** via its REST API.

A case is automatically created containing:

* Alert title and description
* Severity classification
* IOC list (IPs, domains, hashes)
* Full enrichment output
* Original Wazuh alert context

This ensures analysts have complete context from the moment a case is opened.

---

### 6. Automated Response

Depending on the classification, n8n executes one or more of the following response actions:

* **Block IP** via OPNsense firewall API
* **Disable user account** in Microsoft Entra (Azure AD)
* **Revoke active sessions** via Microsoft Graph API
* **Add IOC to MISP** for reuse in future detections

## Example Use Case

### Malicious Login Detection & Response

**Scenario:**
A user successfully authenticates from an IP address with a high AbuseIPDB abuse confidence score.

**Workflow Execution:**

1. Wazuh detects a successful login event and sends the alert to n8n via webhook
2. n8n extracts the source IP from the alert payload
3. VirusTotal and AbuseIPDB return a high-confidence malicious verdict
4. MISP confirms the IP matches a known threat actor infrastructure indicator
5. n8n automatically executes:

   * Disables the user account in Microsoft Entra
   * Revokes all active sessions via Microsoft Graph
   * Blocks the source IP on OPNsense
   * Creates a high-severity case in DFIR IRIS with full enrichment context

Total automated response time: under 30 seconds from alert generation.

## Key Benefits

* Significantly reduces manual SOC workload for routine alert handling
* Enables consistent and repeatable responses with no analyst-to-analyst variation
* Integrates multiple tools into a single unified pipeline
* Highly flexible — new workflows can be added without modifying existing ones
* Open-source and cost-effective

---

## Limitations

* Workflow logic requires careful design to avoid unintended actions (e.g. false positive account lockouts)
* Debugging complex multi-step workflows can be time-consuming
* n8n has no native security focus — SOAR behaviour is achieved through custom workflow design

---

## Design Approach

All workflows are built around the following principles:

* **Automation-first** – routine alerts are handled without manual intervention
* **Fail-safe logic** – if enrichment fails, alerts are escalated rather than silently dropped
* **Modular design** – each workflow step is self-contained and reusable across alert types
* **Scalable** – new integrations can be added without rebuilding existing workflows

---

## Summary

n8n acts as the **central automation engine** of the SOC, transforming static alerts into actionable and automated responses.

It bridges the gap between:

* Detection (Wazuh)
* Intelligence (MISP, VirusTotal, AbuseIPDB)
* Case Management (DFIR IRIS)
* Response (OPNsense, Microsoft Entra)

This enables a fully automated, efficient, and scalable workflow where analysts can focus on complex investigations rather than repetitive alert triage.
