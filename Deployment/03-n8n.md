# n8n — SOAR Automation Setup

## Overview

This guide deploys n8n on the Ubuntu VM at `10.10.10.20` and connects it to Wazuh, MISP, DFIR IRIS, and the external enrichment APIs. By the end of this guide you will have:

* n8n running via Docker Compose with persistent storage
* Credentials configured for all integrations (Wazuh, VirusTotal, AbuseIPDB, URLScan.io)
* The Wazuh webhook integration active — n8n receives alerts in real time
* MISP and IRIS credential placeholders ready (populated after those VMs are deployed)

n8n is the automation layer. It receives every Wazuh alert, enriches it, decides what to do, creates the IRIS case, and triggers the response. Without this, the rest of the stack is read-only dashboards.

## 1. Before You Begin

* Ubuntu 22.04 VM at `10.10.10.20` with hostname `n8n`
* 2 vCPU and 4 GB RAM
* 40 GB disk
* Docker Engine and Compose v2 installed
* Wazuh deployed and healthy (`02-wazuh.md` complete)
* VirusTotal and AbuseIPDB API keys available
* Outbound HTTPS permitted from VLAN 20

## 3. Docker Compose Setup

### 3.1 Create the Project Directory

```bash
mkdir -p ~/n8n && cd ~/n8n
```

### 3.2 Environment File

Create `.env` — this file is never committed to git:

```bash
cat > .env << 'EOF'
# Generate with: openssl rand -hex 32
N8N_ENCRYPTION_KEY=replace_with_64_char_hex_string

N8N_BASIC_AUTH_USER=admin
N8N_BASIC_AUTH_PASSWORD=replace_with_strong_password

GENERIC_TIMEZONE=Europe/London
N8N_HOST=10.10.10.20
WEBHOOK_URL=http://10.10.10.20:5678/
EOF
```

Generate a real encryption key and set a strong password before proceeding:

```bash
openssl rand -hex 32
```

The encryption key protects all stored credentials. If it is lost, all credentials must be re-entered. Store it in a password manager.

### 3.3 Docker Compose File

Create `docker-compose.yml`:

```yaml
services:
  n8n:
    image: docker.n8n.io/n8nio/n8n:latest
    restart: unless-stopped
    ports:
      - "5678:5678"
    environment:
      - N8N_HOST=${N8N_HOST}
      - N8N_PORT=5678
      - N8N_PROTOCOL=http
      - WEBHOOK_URL=${WEBHOOK_URL}
      - GENERIC_TIMEZONE=${GENERIC_TIMEZONE}
      - N8N_ENCRYPTION_KEY=${N8N_ENCRYPTION_KEY}
      - N8N_BASIC_AUTH_ACTIVE=true
      - N8N_BASIC_AUTH_USER=${N8N_BASIC_AUTH_USER}
      - N8N_BASIC_AUTH_PASSWORD=${N8N_BASIC_AUTH_PASSWORD}
      - N8N_METRICS=true
    volumes:
      - n8n_data:/home/node/.n8n
      - ./local-files:/files

volumes:
  n8n_data:
```

> n8n runs HTTP here because it is isolated to VLAN 20 and only accessible from VLAN 10 (Management) via the OPNsense firewall rules. If you expose this beyond the lab, add a TLS-terminating reverse proxy (nginx or Caddy) in front of it.

### 3.4 Start n8n

```bash
docker compose up -d
docker compose logs -f
```

n8n is ready when the logs show `Editor is now accessible via: http://10.10.10.20:5678/`. This takes under 30 seconds on first start.

## 5. Configure Credentials

Credentials in n8n are encrypted at rest using the `N8N_ENCRYPTION_KEY` from `.env`. They never appear in workflow JSON exports. Add them all now — the integration workflows reference them by name.

Navigate to **Settings > Credentials > New Credential** for each entry below.

### 5.1 Wazuh API

| Field | Value |
| ----- | ----- |
| Credential name | `Wazuh API` |
| Type | HTTP Request (Basic Auth) |
| Username | `admin` |
| Password | Your Wazuh API password |
| Base URL | `https://10.10.10.10:55000` |

> The Wazuh API uses a self-signed certificate. In the HTTP Request nodes that use this credential, enable **Ignore SSL Issues** — this is acceptable within an isolated lab VLAN.

## 6. Wazuh Webhook Integration

n8n receives alerts from Wazuh via webhook. The webhook URL is the public endpoint n8n exposes, and Wazuh is configured to POST alerts to it.

### 6.1 Get the Webhook URL

In n8n, create a new workflow. Add a **Webhook** node:

* HTTP Method: `POST`
* Path: `wazuh-alerts`
* Authentication: None (the endpoint is isolated to VLAN 20)
* Response Mode: Immediately

The resulting webhook URL is:

```
http://10.10.10.20:5678/webhook/wazuh-alerts
```

Activate the workflow (toggle in the top-right corner). The webhook only receives events when the workflow is active.

### 6.2 Configure Wazuh to Forward Alerts

On the Wazuh VM (`10.10.10.10`), add an integration block to the Wazuh Manager configuration:

```bash
docker exec -it single-node-wazuh.manager-1 bash
```

Append the following to `/var/ossec/etc/ossec.conf` inside the `<ossec_config>` block, or add it to a new file in `/var/ossec/etc/ossec.conf.d/`:

```xml
<ossec_config>
  <integration>
    <name>custom-webhook</name>
    <hook_url>http://10.10.10.20:5678/webhook/wazuh-alerts</hook_url>
    <level>7</level>
    <alert_format>json</alert_format>
  </integration>
</ossec_config>
```

`<level>7</level>` forwards alerts with severity 7 or above. Adjust this threshold after observing alert volume — setting it too low will flood n8n with low-priority events.

Restart the manager to apply:

```bash
docker restart single-node-wazuh.manager-1
```

### 6.3 Verify Alert Delivery

Generate a test alert from the Wazuh dashboard (**Tools > API Console**) or trigger a failed SSH login on the test endpoint. In n8n, navigate to the Webhook node's execution log — you should see the incoming JSON payload within seconds.

## 7. Alert Payload Structure

Understanding the Wazuh alert JSON structure is essential for building enrichment workflows. A typical alert contains:

```json
{
  "id": "1700000000.12345",
  "timestamp": "2025-01-15T10:23:44.123+0000",
  "rule": {
    "id": "60106",
    "level": 10,
    "description": "Multiple authentication failures"
  },
  "agent": {
    "id": "001",
    "name": "victim-endpoint",
    "ip": "10.10.20.101"
  },
  "data": {
    "srcip": "203.0.113.42",
    "srcuser": "Administrator"
  },
  "location": "WinEventLog"
}
```

The key fields used in enrichment workflows are:
* `rule.id` — determines which enrichment path to follow
* `rule.level` — used for triage priority
* `data.srcip` — submitted to VirusTotal and AbuseIPDB
* `agent.name` — used to identify the affected endpoint in IRIS cases

## 8. n8n Workflow Structure

The full automation workflows are imported in `06-integration.md`. The structure each workflow follows is:

```text
Webhook (receive alert)
    │
Set (extract rule.id, srcip, agent.name)
    │
Switch (route by rule type)
    │
HTTP Request (VirusTotal lookup on srcip / file hash)
    │
HTTP Request (AbuseIPDB lookup on srcip)
    │
IF (malicious confidence > threshold)
    │
    ├── HTTP Request (create IRIS case)
    │
    ├── HTTP Request (add IOC to MISP)
    │
    └── HTTP Request (block IP via OPNsense API)
```

Do not build these nodes manually — the JSON exports in `06-integration.md` import them in one step.

## 9. OPNsense API Credential (Response Action)

n8n calls the OPNsense API to block IPs as an automated response action. OPNsense must have the API enabled and a key generated.

In OPNsense, navigate to **System > Access > Users**. Create a new user:

| Field | Value |
| ----- | ----- |
| Username | `n8n-api` |
| Password | Strong password |
| Group | Admins (for firewall rule write access) |

After saving, click **Edit** on the user and generate an API key. Download the key file — it contains the key and secret. Store both values.

Add to n8n credentials:

| Field | Value |
| ----- | ----- |
| Credential name | `OPNsense API` |
| Type | HTTP Header Auth |
| Name | `Authorization` |
| Value | `Basic base64(key:secret)` |

The base64 value is generated with:

```bash
echo -n "key:secret" | base64
```
